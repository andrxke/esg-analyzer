## Problem Statement

Companies publish glossy ESG / sustainability reports full of confident claims ("net-zero
by 2030," "industry-leading emissions reduction," "carbon neutral"). A lot of that is
greenwashing: bold language with no baseline, no interim milestones, missing Scope 3
emissions, offset-heavy "neutrality," or targets that quietly vanish between one year's
report and the next. There is no fast, glanceable way for a normal person to tell a
serious sustainability effort from marketing spin.

The analyzer reads a company's own published sustainability report and produces a
Consumer-Reports-style verdict — **Recommended / Improving / Not Recommended** — backed
by receipts: the company's own claims quoted next to the internal tells that undercut them.

## What Makes This Cool

The "whoa" is the **claims-vs-reality gap surfaced from a single document**. Most
greenwashing is self-incriminating — the evidence is already inside the report. You don't
need a scraper zoo to catch it. The killer feature is **goalpost-shifting detection**:
diff this year's report against last year's and catch the 2025 target that got quietly
deleted. It's a screenshot people forward. The label is the hook; the receipts are the
proof that make it believable and shareable.

## Constraints

- **Near-zero cost is a hard requirement.** Sustained by student/free tiers + modest ad
  revenue. The only real cost driver is LLM calls, so LLM usage must be bounded by design.
- **Side project scope.** Weekend-to-a-few-weekends build, not a six-month slog.
- **Legal safety.** Real, named public companies appear next to "Not Recommended." Verdicts
  must read as analysis/criticism that quotes the source, never as a factual accusation
  invented by the tool.
- **Self-sufficiency before partnerships.** Ads/newsletter should cover costs first;
  sponsorships/affiliates/partnerships come later once there's an audience.

## Approaches Considered

### Approach A: Curated DB + rate-limited live analyzer (CHOSEN)
- Summary: Batch-analyze a curated set of companies offline (you control the count/cost).
  Cache results and serve them as fast static verdict pages — free to browse, great for SEO
  and sharing. A "paste your own report" box exists but is rate-limited and cached so live
  LLM cost stays bounded. Email capture on every page.
- Effort: M
- Risk: Low
- Pros: browsing costs ~$0 regardless of traffic; builds the full stack (extraction, LLM
  prompting, serverless, DB, frontend); growing public DB = SEO + newsletter fuel.
- Cons: live user analysis needs rate limiting; you seed the initial company list by hand.
- Reuses: Vercel/Cloudflare free tier, a free/cheap LLM tier, free-tier Postgres/SQLite,
  an email tool's free plan.

### Approach B: Fully dynamic "analyze any report" app
- Summary: Anyone pastes any report; backend runs the LLM live and stores it.
- Effort: L / Risk: Med
- Pros: feels magical, unbounded coverage, most technically ambitious.
- Cons: unbounded per-analysis LLM cost (the #1 threat to zero-cost), abuse/spam surface,
  prompt-injection risk from uploaded documents, variable quality on random uploads.

### Approach C: Newsletter-first, analyzer as a private tool
- Summary: Analyzer is a script you run; you hand-pick damning analyses and publish them as
  a static site + email list.
- Effort: S / Risk: Low
- Pros: ships in a weekend, ~$0, fastest path to an audience, highest editorial quality.
- Cons: more blog than product, least interactive-frontend skill-building, manual publishing.

## Recommended Approach

**Approach A.** It hits all three goals at once: real full-stack skill-building
(PDF/text extraction → LLM prompt engineering → serverless API → DB → frontend), near-zero
cost by design (cache-first browsing, rate-limited live analysis), and a growing public
verdict database that both ranks in search and feeds the weekly newsletter. It's the honest
middle between C (ships fast but is really a blog) and B (impressive but risks runaway cost
and abuse).

A sensible sequencing note: you can borrow C's speed by shipping a small curated batch
first (even 10–20 companies), then layer the live analyzer box on top. That gets a public,
shareable thing live fast without waiting on the interactive path.

## Architecture Sketch (v1)

- **Ingest:** accept a report as a PDF upload or a URL. Extract text (a PDF text-extraction
  library; fall back to OCR only if needed later).
- **Analyze:** one grounded LLM pass over the extracted text. The prompt forces the model to
  (a) quote specific claims verbatim, (b) map each claim to a defined "tell" from a fixed
  rubric, (c) return structured JSON (label + findings + quotes), never freeform prose.
  Document content is treated strictly as untrusted data, not instructions (prompt-injection
  defense). Grounding in the single document is what keeps hallucination and legal risk low.
- **Verbatim-quote verification gate (locked 2026-07-10):** after the LLM returns findings,
  every quote is verified as an **exact substring of the extracted source text** (whitespace-
  normalized). Any finding whose quote does not match verbatim is dropped (and the analysis is
  flagged for manual review). The LLM proposes; deterministic code verifies against the source.
  This stops *fabricated quotes* — but note (per outside voice) the quote is NOT the main legal
  surface. See the label-framing requirement below.
- **Label-as-opinion framing (locked 2026-07-10) — the real legal surface.** The label
  ("Not Recommended") is the tool's own conclusion published next to a named company — that, not
  the quotes, is what a company would dispute. v1 requirements:
  - Every verdict page shows the **rubric/criteria** the label was derived from.
  - The label is framed as **"our assessment against these criteria,"** tied to the on-page
    evidence — never asserting intent ("greenwashing," "lying") as fact.
  - A plain-language **methodology + disclaimer page** states this is opinion/analysis based on
    published criteria applied to the company's own report.
  This is the Consumer-Reports/Wired defense: a transparent opinion from stated criteria, not a
  factual accusation. Mostly presentation — cheap insurance for putting labels next to real names.
- **Rubric (the tells):** no baseline for a claim; target with no interim milestones; Scope 3
  missing while Scope 1/2 are loud; "carbon neutral" that's mostly purchased offsets;
  vague superlatives with no number; and — the flagship — goalpost shifting (a target present
  last year, absent or weakened this year).
- **Store & serve (v1, locked 2026-07-10):** the analyzer writes each verdict to a
  **build-time SQLite file** (no runtime DB, no host/DB coupling to decide). The SSG build
  reads SQLite at build time and emits **static pages** — browsing hits the CDN, never the
  LLM or a live DB. Every page carries an email-capture box.
- **Company-request intake (v1):** a **free third-party form** (Tally / Formspree / Google
  Form) collects "please analyze Company X" requests into a dashboard you read when deciding
  what to analyze next. This is the one request-time write, deliberately offloaded to a free
  service so the site itself stays fully static and $0. Decoupled from the verdict store.
- **Cost control:** curated analyses run offline in a batch you trigger (deploy-on-push is a
  separate thing from the batch run — deploying does not re-analyze). No live LLM calls in v1.
  (The rate-limited public paste box + its KV store move to the deferred fast-follow.)
- **Incremental batch (locked 2026-07-10):** content-hash each source report and store the hash
  with the verdict. A batch run only analyzes reports that are new or whose source changed —
  re-running is idempotent and free, adding one company costs exactly one LLM analysis. This
  keeps cost flat as coverage grows (direct lever on the "control how many LLM calls" model).

### Pipeline failure handling (locked 2026-07-10)

The pipeline is ingest → extract → LLM → verify → store. Each stage has an explicit failure rule:

- **Extraction guard:** validate extracted text length/quality before the LLM call. Below a
  threshold (scanned-image PDF, bad encoding, near-empty) → mark the report `extraction-failed`,
  skip the LLM, do not produce a verdict.
- **Schema validation + one retry:** LLM output is validated against a strict JSON schema. On
  malformed/non-conforming output, retry once; on second failure, mark `analysis-failed` and move on.
- **`insufficient-data` is a distinct state:** zero findings does NOT auto-map to "Recommended."
  A report that's clean vs. one that's too thin/truncated to judge get different states; never
  silently label Recommended on absence of findings.
- **Per-report failure isolation:** one bad report marks itself failed and the batch continues —
  a single failure never kills the whole run.

### LLM cost model (the make-or-break assumption)

The whole zero-cost claim rests on "one report = one cheap call." That is NOT safe by default:
a full sustainability report is 50–200 pages, easily 100k+ tokens, which overflows cheap
context windows and blows the budget in one call. So the design does **not** feed the whole
report to the model. Instead:

- **Two-tier detection (locked 2026-07-10) — resolves the pre-filter vs. omission-tell collision.**
  Half the tells are about what's *absent* (no Scope 3, no baseline, no interim milestones), and you
  cannot detect an omission from an excerpt — the model can't tell "the company omitted this" from
  "the pre-filter dropped it." So:
  - **Tier 1 — presence scan over the WHOLE report (cheap, mostly no LLM):** for the omission tells,
    check the full extracted text for presence of the thing (does "Scope 3" appear anywhere? is there
    a baseline year? any interim milestone?). This is regex/keyword presence scanning over the full
    document — near-free even on 100k+ tokens because it's not an LLM call (or a single yes/no
    presence prompt if a tell truly needs the model).
  - **Tier 2 — filtered LLM pass for quoted evidence:** send only the claim/number sections
    (~15–30k token cap) to the LLM for the detailed quoted-evidence findings on the presence tells.
  This keeps LLM cost bounded (Tier 2 is filtered) while making omission detection correct (Tier 1
  sees the whole document).
- **Validate first (Next Step #2).** Before building the web layer, measure real token counts
  on 3 actual reports. If a filtered report doesn't fit the budget cheaply, the cost model is
  wrong and must be revisited — this is the single most important thing to prove early.

### Goalpost-shifting: structured extraction + programmatic diff (NOT "spot what's missing")

LLMs are unreliable at noticing omissions — they hallucinate additions less than they miss
deletions. So the flagship feature is **not** "ask the model what changed." Instead:

- For each report edition, the LLM extracts a **normalized list of targets** (metric, value,
  baseline year, target year) as structured JSON — one edition at a time.
- A plain, deterministic **code diff** compares this year's target list against last year's.
  A target present last year and absent/weakened this year is flagged programmatically, not
  by the model.
- This requires **two manually-sourced report editions** per curated company. **Deferred to
  fast-follow #2** (see v1 Scope + NOT in Scope) — this design is recorded here so it's ready
  to build, but it is NOT part of v1.

  fit the ~15–30k budget on real reports (Next Step #3 measurement).
- **Label thresholds:** what exactly moves a company from Improving to Not Recommended? Needs
  a written, defensible rubric so verdicts are consistent and explainable. (Next Step #1.)
- **Report freshness / sourcing:** record which report edition (publisher, year, URL, date
  retrieved) each verdict is based on, and show it on the page — protects you legally and
  builds trust. Especially important for the two editions the goalpost diff depends on.
- **Seed list:** which first 10–20 companies to analyze (pick recognizable names with
  public, multi-year reports so the goalpost feature has data and pages are shareable).

## Test Plan (locked 2026-07-10)

v1 ships tests alongside the code — target 100% of the 26 enumerated pipeline/UI paths. Highlights:

- **Verbatim-quote gate (legal-critical):** tests for quote-matches-source → keep,
  quote-not-in-source → drop finding, and all-findings-dropped → `insufficient-data`.
- **Pipeline failure paths:** extraction-below-threshold → `extraction-failed`; malformed LLM
  JSON → retry → `analysis-failed`; over-token-cap → truncation; no claim sections found.
- **Label logic:** findings → correct label; zero findings → `insufficient-data`, never Recommended.
- **SSG build:** verdict renders to page; failed-state reports excluded from public pages.
- **Email capture:** valid submits, invalid/empty rejected, duplicate handled.
- **Golden-set eval (`[→EVAL]`):** 3–5 real reports with known-correct labels + expected tells,
  re-run on every prompt change. This is the only test that guards label *quality* (mocked unit
  tests prove plumbing, not verdict correctness) and it doubles as the Next-Step-#2 cost-validation
  harness (measure real token counts on the same reports).
- **One E2E flow:** landing → verdict page loads → label + quotes + source citation visible.

## Parallelization Strategy

| Step | Modules | Depends on |
|------|---------|-----------|
| Rubric doc | docs | — |
| Analyzer script | analyzer/ | Rubric doc |
| Golden-set eval | eval/ | Analyzer script |
| Web layer (SSG + pages + form) | site/ | Analyzer script (verdict output shape) |

`Lane A: rubric → analyzer → eval (sequential, each depends on prior)`
`Lane B: web layer scaffold (site/) can start in parallel once verdict JSON shape is fixed`

Execution order: rubric first (blocks everything). Then analyzer. Once the verdict JSON shape is
frozen, the eval (Lane A) and the web-layer scaffold (Lane B) can run in parallel worktrees — they
touch different modules (`eval/` vs `site/`). Merge, then ship the curated batch. Mostly sequential
early (the rubric gates all), with one real parallel window (eval + site) after the schema freezes.
